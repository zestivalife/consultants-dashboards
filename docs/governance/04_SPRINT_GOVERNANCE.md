# 04 Sprint Governance

Document ID: EDO-04  
Version: 1.0  
Status: ACTIVE  
Owner: Enterprise Delivery Office  
Review Frequency: Every sprint

---

## Purpose

Define how sprint work is planned, executed, reviewed and improved.

## Scope

Applies to product, platform, engineering, QA, documentation and release work.

## Sprint Operating Model

| Activity | Required Outcome |
|---|---|
| Sprint Planning | Goals, scope, risks and readiness are agreed |
| Backlog Refinement | Upcoming work meets Definition of Ready |
| Architecture Review | Material changes are reviewed before coding |
| Daily Development | Blockers and regressions are surfaced early |
| Code Review | Changes meet engineering review standards |
| QA | Acceptance criteria and regression checks are verified |
| Demo | Product Owner can see working behavior or reviewed documentation |
| Retrospective | Improvements and root causes are captured |
| Documentation Updates | Product Bible, registry, changelog and ADRs stay aligned |
| Release Planning | Deployment, rollback and verification are prepared |

## Inputs

- Roadmap.
- Product Bible.
- Ready backlog.
- Delivery priorities.

## Outputs

- Sprint goal.
- Accepted work.
- Updated documentation.
- Retrospective actions.

## Dependencies

- Definition of Ready.
- Definition of Done.
- Code Review Standard.
- Testing Strategy.

## Success Metrics

- Sprint work aligns to roadmap.
- Fewer unplanned architecture pivots.
- Product Owner can validate completed work.

## Risks

- Starting unready work.
- Missing documentation updates.
- Demoing static UI instead of working workflows.
